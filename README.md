# Полиморфные абстрактные типы данных (Лабораторная работа №3 - Lazy Collections, Streams & Finite State Machine)

Расширение библиотеки **Sequences** линейных структур данных. В рамках данного проекта с нуля разработаны механизмы **ленивых вычислений (Lazy Evaluation)**, **мемоизации**, **трансфинитной математики**, **полиморфных потоков (Streams)**, а также реализован **Конечный Автомат (FSM)** с полноценным графическим интерфейсом на базе **ImGui + SFML**. 

---

## Архитектура и основная логика

### 1. Управление памятью: Паттерн Арена (`LazyContext`)
Поскольку функциональное программирование подразумевает создание множества промежуточных иммутабельных объектов при цепочках вызовов (например, `seq->where(...)->concat(...)`), был разработан собственный менеджер памяти. Любые созданные генераторы и ленивые списки регистрируются в `LazyContext`. При выходе контекста из области видимости он автоматически очищает всю память.


### 2. Ленивая последовательность (`LazySequence<T>`)
Неизменяемая структура данных, элементы которой вычисляются только "по требованию".
*   **Мемоизация (кэширование):** Элементы кэшируются во внутреннем хранилище (`mutable MutableArraySequence`). Повторное обращение к элементу по индексу происходит за **$O(1)$** и не вызывает повторной генерации.
*   **Трансфинитные числа (`Cardinal`):** Ленивые коллекции могут быть бесконечными (ряд Фибоначчи, математические функции). Использование класса `Cardinal` предотвращает зависание программы (Infinite Loop) при попытке вызова `get_length()` у бесконечного списка.

### 3. Генераторы и Функциональная комбинаторика
Движок ленивости полностью абстрагирован в семейство классов `Generator<T>`.
*   **Stateless / Stateful:** Поддерживается генерация по формуле (`FunctionGenerator`) и генерация на основе рекуррентного правила с использованием паттерна **"Скользящее окно" (`RecurrentGenerator`)**.
*   **Комбинаторика вместо дублирования:** Сложные операции модификации (`insert_at`, `remove_at`, `append`) реализованы не через копирование памяти, а через функциональную комбинацию **базовых примитивов** (`SliceGenerator`, `ConcatGenerator`, `SingleItemGenerator`). Это позволяет "вырезать" и "склеивать" куски бесконечных множеств в реальном времени, сохраняя иммутабельность и работая за **$O(1)$** по памяти.

### 4. Полиморфные Потоки данных (`Streams`)
Реализована строгая инверсия зависимостей (Dependency Inversion) на основе интерфейсов `IReadOnlyStream` и `IWriteOnlyStream`.
*   Потоки физически уничтожают прочитанные данные (отсутствует кэш).
*   Контракт поддерживает проверку на возможность перемотки (`is_can_seek`). Потоки из массивов (`SequenceReadStream`) поддерживают перемотку, потоки "на лету" (`GeneratorReadStream`) двигаются только вперед.
*   Реализован паттерн "Адаптер" (`ReadWriteStream`) для бесшовной склейки источников чтения и записи.

## Тематическая задача: Конечный Автомат (FSM)

В качестве прикладной задачи (согласно варианту №6) реализована абстрактная машина состояний.
*   **Матрица переходов (`FsmRuleEngine`):** Позволяет задавать правила вида `(Состояние, Событие) -> Новое Состояние`. Поддерживает Fallback-состояния для защиты от "мусорных" данных.
*   **Потоковая генерация (`FsmGenerator`):** Сам автомат спроектирован как ленивый генератор. Он принимает на вход абстрактный поток событий `IReadOnlyStream`, перерабатывает его и отдает поток измененных состояний. 
*   **Стресс-устойчивость:** Автомат способен обработать огромное количество событий за считанные секунды с потреблением оперативной памяти $\approx O(1)$, так как элементы создаются, обрабатываются и уничтожаются по одному, без загрузки в ОЗУ.

## Графический интерфейс (ImGui + SFML)

Для визуализации работы реализовано полноценное оконное Desktop-приложение (везде задействованы ADT Sequence).

*   **Ручной режим (Manual Mode):** Интерактивная панель, позволяющая отправлять события в Автомат по нажатию кнопок. Содержит плавный скроллируемый лог переходов, возможность копирования лога в буфер обмена и динамическое масштабирование интерфейса (UI Scale) под любые мониторы.
*   **Автоматический стресс-тест:** Инструмент для бенчмаркинга. Позволяет сгенерировать поток от 1 тысячи до 100 млн событий на лету. Встроен механизм Anti-Freeze (UI Pump), гарантирующий, что приложение не "зависнет" со стороны ОС (Windows Not Responding) во время массивных вычислений.

## Руководство по сборке

### Требования
*   **Компилятор:** Поддерживающий стандарт **C++23** (GCC, Clang, MSVC).
*   **Система сборки:** CMake версии **3.20** или выше.
*   Зависимости (SFML, Dear ImGui, Google Test) конфигурируются автоматически через `FetchContent` или из локальной папки `external/`.

### Сборка через консоль
1. Клонируйте репозиторий и перейдите в папку:
   ```bash
   git clone <URL_репозитория>
   cd PADT-Lab4-LazySequences
   ```
2. Соберите проект:
   ```bash
   cmake --build build --config Release
   ```

3. Запуск: 
   ```bash
   ./build/app/Debug/Lab4_app.exe 
   ```
### Сборка в IDE (Visual Studio / CLion / VS Code)
Проект настроен на парадигму **Modern CMake**. Просто откройте папку проекта в вашей IDE. 
Вам будут доступны несколько независимых Target'ов:
*   `Lab4_app` — Графическое оконное приложение (Автомат состояний).
*   `tests_sequences` — Тесты математики и ленивых коллекций.
*   `tests_streams` — Тесты полиморфных потоков.
*   `tests_fsm` — Тесты автомата состояний.

## Запуск тестов

Тестовое покрытие включает граничные случаи (Edge Cases), проверку утечек памяти в `LazyContext` (через классы-шпионы) и защиту от вечных циклов.

Запуск всех тестов разом через CTest:
```bash
cd build
ctest -C Release --output-on-failure
```

Запуск тестов по-отдельности через GTest:
```bash
./build/tests/Debug/tests_fsm.exe 
./build/tests/Debug/tests_sequences.exe 
./build/tests/Debug/tests_streams.exe 
```
